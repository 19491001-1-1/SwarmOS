# v0.9.3 — Web Login & Session Protection

> 本文档是 v0.9.2 后的 Web 安全增强版本规划，可直接交给 Claude Code / Codex 执行。执行时从 `main` 新建 `feat/v0.9.3-web-login-auth` 分支，按 TDD 推进，完成后运行 `pnpm verify`；如果触及 Cloudflare Worker，再运行 wrangler dry-run。

## 目标

确保 Crewden Web 页面不能被未登录用户直接访问，尤其是公网 Cloudflare Pages + Cloudflare Hub 部署时。

v0.9.3 不是完整多用户系统，而是先做单用户/小团队可用的登录态保护：

1. 打开 Web UI 时，如果没有有效登录态，显示登录页。
2. 登录成功后才能进入 Workspace。
3. REST API 和 WebSocket 都继续带认证 token。
4. token 不再必须在构建时写死到 `VITE_WEB_AUTH_TOKEN`。
5. 支持退出登录。
6. 本地开发和 Cloudflare 公网部署都有清晰配置方式。

## 当前基线

已经存在：

- `WEB_AUTH_TOKEN`：Cloudflare Worker 侧用于保护 `/api/*` 和 `/ws`。
- `VITE_WEB_AUTH_TOKEN`：Web build 时注入，`packages/web/src/api.ts` 会自动加：
  - REST：`Authorization: Bearer <token>`
  - WebSocket：`/ws?token=<token>`
- Cloudflare 测试已覆盖无 token / 错 token / 正确 token。

当前不足：

- Web 页面没有登录页。
- 如果 `VITE_WEB_AUTH_TOKEN` 构建进前端，token 等于被静态发布出去，不适合公网长期使用。
- 用户无法在浏览器里输入 token 登录。
- 用户无法退出登录。
- token 过期/错误时，UI 没有统一回到登录页。
- 本地 server 默认不强制 browser auth，公网模式和本地模式语义不统一。

## 非目标

- 不做多用户账号体系。
- 不接 OAuth / GitHub / Google 登录。
- 不做 RBAC。
- 不做密码重置。
- 不做 refresh token。
- 不把 daemon API key 和 browser token 混用。

## 安全原则

- Browser token 只用于浏览器访问 `/api/*` 和 `/ws`。
- Daemon 继续使用 `DAEMON_API_KEY`，不受本版本影响。
- Agent internal API 继续使用 per-agent token，不受本版本影响。
- Web 登录 token 不应打印到 console 或日志。
- 不把真实 token 写入仓库。
- 对公网部署，未配置 `WEB_AUTH_TOKEN` 必须 fail closed。

## 产品行为

### 1. 登录页

未登录时显示简洁登录页：

- 标题：Crewden
- 输入框：Access token
- 按钮：Sign in
- 错误提示：Invalid token / Server unavailable
- 可选说明：Ask the workspace owner for an access token.

要求：

- 登录页不加载 workspace 数据。
- 登录页不打开 WebSocket。
- 登录错误不能泄漏 server 细节。
- 支持 Enter 提交。

### 2. 登录成功

流程：

1. 用户输入 token。
2. Web 调用 auth check endpoint。
3. 成功后把 token 保存到浏览器本地。
4. 进入 Workspace。
5. 后续 REST/WS 使用该 token。

存储建议：

- MVP 使用 `localStorage`，key 例如 `crewden.webAuthToken`。
- 可选 `sessionStorage` 模式后续再做。
- 所有 token 读取集中在 auth 模块，不要散落在组件里。

### 3. 退出登录

UI 提供 Sign out：

- 清除本地 token。
- 关闭 WebSocket。
- 清空当前页面敏感状态或刷新到登录页。

### 4. 认证失败处理

如果任意 API 返回 401：

- 清除本地 token。
- 关闭 WebSocket。
- 回到登录页。
- 显示 session expired / invalid token。

如果 WebSocket 被 401/Unauthorized 拒绝：

- 回到登录页。

## API 设计

### 推荐新增 endpoint

```http
GET /api/auth/whoami
Authorization: Bearer <WEB_AUTH_TOKEN>
```

成功：

```json
{
  "authenticated": true,
  "mode": "token"
}
```

失败：

```json
{
  "error": "Unauthorized"
}
```

原因：

- Web 可以用它验证输入 token。
- 不需要暴露用户信息。
- 本地 server 和 Cloudflare Worker 可以实现同一接口。

### Server 本地模式

当前 local server 不强制 browser auth。v0.9.3 建议统一：

- 新增 `WEB_AUTH_TOKEN` 环境变量。
- 如果设置了 `WEB_AUTH_TOKEN`，local server 也保护 `/api/*` 和 `/ws`。
- 如果未设置：
  - dev 模式可允许匿名访问；
  - production-like 启动应给出警告或拒绝，具体由实现选择，但必须文档化。

### Cloudflare 模式

Cloudflare Worker 已有 browser auth 机制，v0.9.3 需要补：

- `GET /api/auth/whoami`
- 保持 `/api/*` 和 `/ws` 强制 token。
- `WEB_AUTH_TOKEN` 未配置时继续 fail closed。

## Web 技术计划

### Task 0：分支与基线

```bash
git switch main
git pull --ff-only
git switch -c feat/v0.9.3-web-login-auth
pnpm verify
```

### Task 1：Web auth 模块

文件：

- `packages/web/src/auth.ts`
- `packages/web/src/api.ts`

工作：

- 新增 auth token storage：
  - `getStoredAuthToken()`
  - `setStoredAuthToken(token)`
  - `clearStoredAuthToken()`
- `api.ts` 不再只依赖 build-time `VITE_WEB_AUTH_TOKEN`。
- token 来源优先级：
  1. localStorage login token
  2. `VITE_WEB_AUTH_TOKEN` fallback（用于本地/CI/临时部署兼容）
- WebSocket URL 使用同一 token 来源。

验收：

- 无 token 时 API 不带 Authorization。
- 登录 token 存在时 API 带 Authorization。
- build-time token 仍可兼容旧部署。

### Task 2：登录页与 Auth Gate

文件：

- `packages/web/src/App.tsx`
- 新增 `packages/web/src/components/LoginView.tsx`

工作：

- App 启动时先检查 token。
- 无 token 显示 LoginView。
- 有 token 调用 `/api/auth/whoami` 验证。
- 验证成功后加载 workspace 数据和 WebSocket。
- 验证失败回到 LoginView。

验收：

- 未登录时不请求 channels/agents/tasks。
- 登录成功后正常进入主界面。
- token 错误时显示错误。

### Task 3：统一 401 处理

文件：

- `packages/web/src/api.ts`
- `packages/web/src/App.tsx`

工作：

- fetch wrapper 统一处理 401。
- 401 触发 auth expired callback 或抛出专用错误。
- App 捕获后清 token、关 WebSocket、回登录页。

验收：

- token 被服务端改掉后，下一次 API 调用会回登录页。
- WebSocket auth 失败也会回登录页。

### Task 4：退出登录

文件：

- `Sidebar.tsx` 或 workspace selector 相关组件
- `App.tsx`

工作：

- 在 workspace/user 菜单中增加 Sign out。
- 清除 token。
- 关闭 WebSocket。
- 回到登录页。

验收：

- 点击 Sign out 后无法继续看到 workspace 数据。
- 刷新后仍在登录页。

### Task 5：Server auth endpoint

文件：

- `packages/server/src/app.ts`
- 新增或修改 routes
- `packages/server/test/*`

工作：

- 实现 `GET /api/auth/whoami`。
- 如果 `WEB_AUTH_TOKEN` 配置，校验 bearer token。
- 可选：为 local server 的 `/api/*` 和 `/ws` 增加 browser auth middleware。
- 保持 daemon/internal agent auth 不受影响。

测试：

- 无 token 请求 whoami 返回 401。
- 错 token 返回 401。
- 正确 token 返回 authenticated。
- 未配置 `WEB_AUTH_TOKEN` 的 dev 模式行为按设计测试。

### Task 6：Cloudflare endpoint 对齐

文件：

- `packages/cloudflare/src/index.ts`
- `packages/cloudflare/test/hub.test.ts`

工作：

- 新增 `/api/auth/whoami`。
- 测试正确/错误 token。
- 确认已有 `/api/*` 和 `/ws` auth 不回退。

验收：

```bash
pnpm --filter @crewden/cloudflare test
```

### Task 7：部署文档与脚本

文件：

- `.env.example`
- `docs/cloudflare-central-hub.md`
- `scripts/deploy-cloudflare-pages.sh` 如需要

工作：

- 说明 `WEB_AUTH_TOKEN` 和 `VITE_WEB_AUTH_TOKEN` 的区别：
  - `WEB_AUTH_TOKEN`：服务端/Worker 校验用 secret。
  - `VITE_WEB_AUTH_TOKEN`：兼容旧模式，不推荐公网长期写死。
- Pages 部署不应强制要求 `VITE_WEB_AUTH_TOKEN`，因为用户可以登录输入 token。
- 仍允许 CI/私有部署注入 `VITE_WEB_AUTH_TOKEN` 跳过登录。
