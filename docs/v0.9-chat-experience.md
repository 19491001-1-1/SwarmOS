# v0.9 — Chat Experience, Presence & Threads

> This document is an implementation-ready roadmap. Claude Code or another coding agent should execute it from a fresh branch, run tests after each meaningful slice, and continue until all acceptance checks pass.

## Goal

Make the group chat feel like a mature collaboration surface instead of a raw message log.

v0.9 focuses on three user-visible upgrades:

1. Show each participant's current availability/work state directly on avatars.
2. Render group chat messages with polished Markdown and readable mention chips.
3. Add thread support so a topic can be discussed in depth without flooding the main channel.

The result should feel closer to Slack: easy scanning in the main channel, clear status at a glance, and focused side conversations for deep dives.

## Non-Goals

- Do not redesign the whole application shell.
- Do not add cross-channel thread search in v0.9.
- Do not implement unread counters for threads yet unless needed for local UI state.
- Do not make agent DMs thread-aware in this version.
- Do not introduce rich text editing; composer stays plain text with Markdown support.

## Current Baseline

- `messages` currently store only `id`, `channelId`, `senderName`, `content`, `agentId`, `createdAt`.
- `ChannelView` renders plain text with `whiteSpace: pre-wrap`.
- `Composer` can target one running agent via a select, but typed `@name` is not parsed or rendered.
- Agent status exists as `inactive | starting | running | working | idle | error`.
- Activity events exist and can be used to derive more specific "thinking/working/output/error" hints.
- There is no message thread model.

## Product Requirements

### 1. Avatar Presence

Every visible message avatar should have a small status indicator.

Status mapping:

| Agent/User state | Dot style | Tooltip label |
|---|---|---|
| `working` | animated blue/purple ring or pulsing dot | Working |
| latest activity `thinking` | animated blue/purple ring or pulsing dot | Thinking |
| `running` | solid green dot | Online |
| `idle` | muted green/gray dot | Idle |
| `starting` | small spinner/ring | Starting |
| `error` | red dot | Error |
| `inactive` or missing | gray hollow dot | Offline |
| human user | solid green dot if browser session is active, otherwise neutral | You |

UI constraints:

- Indicator should sit on the lower-right of the avatar.
- It must be readable but not noisy.
- It should not change avatar dimensions or shift message rows.
- For grouped messages without avatar, do not repeat status dots.
- In agent side panels, reuse the same visual language where possible.

Implementation hint:

- Add a reusable component such as `PresenceAvatar` or `AgentAvatar`.
- Inputs should be `name`, `isAgent`, `status`, and optional latest activity.
- Keep deterministic avatar visuals, but reduce heavy borders if needed to make Markdown content calmer.

### 2. Message Visual Polish

The channel message list should support compact scanning and rich content.

Required improvements:

- Preserve Slack-like grouping by sender and time window.
- Render Markdown safely:
  - paragraphs
  - bold/italic/inline code
  - fenced code blocks
  - bullet and numbered lists
  - blockquotes
  - tables if library support is cheap
  - links with `target="_blank"` and `rel="noreferrer"`
- Do not render raw HTML from message content.
- Add clear styling for agent answers:
  - readable line height
  - code blocks with horizontal scroll
  - tables with subtle borders
  - no huge margins inside messages
- Keep messages dense enough for operational chat.
- Add hover actions, at minimum:
  - create task
  - reply in thread
  - copy link or copy text if cheap

Recommended libraries:

- `react-markdown`
- `remark-gfm`

If adding dependencies is not desirable, implement a small safe renderer for mentions and code fences only, but prefer proven Markdown libraries.

### 3. Mentions

Mentions should be visible and machine-readable enough for later routing.

Supported syntax:

- `@name`
- `@displayName` when it has no spaces
- Mentions inserted by composer's existing agent selector should still work.

Rendering:

- Matched agents render as compact mention chips.
- Unknown mentions render as normal text or a muted chip, but must not break layout.
- Mention chips should show display name when known.
- User mention `@user` should also render distinctly.

Data model:

Add optional mention metadata to messages:

```ts
mentions?: Array<{
  type: 'agent' | 'user';
  id: string;
  label: string;
}>;
```

Storage options:

- Preferred: add `mentions` JSON text column to `messages`.
- Acceptable for v0.9: compute mentions at read time from content plus current agents. Persisting is better because names may change.

Composer behavior:

- Keep current plain textarea.
- When user types `@`, show a lightweight suggestion menu for agents and `@user`.
- Selecting a suggestion inserts the mention text.
- This can be implemented after rendering if time is tight, but mention rendering is required.

### 4. Threads

Threads allow focused discussion under a root message.

User stories:

- As a user, I can click "Reply in thread" on a channel message.
- A right-side thread panel opens with the root message at the top.
- I can send replies in that thread.
- Thread replies do not appear as full messages in the main channel.
- The root message in the main channel shows a thread summary such as `3 replies · latest 10:42`.
- Clicking the summary reopens the thread.
- Thread messages support the same Markdown and mentions rendering.
- WebSocket updates should update the thread panel and root summary in real time.

Data model:

Add fields to `messages`:

```ts
threadRootId?: string; // undefined/null for top-level messages
replyCount?: number;   // derived for root messages
latestReplyAt?: string; // derived for root messages
mentions?: Mention[];
```

Rules:

- A root message has `threadRootId = null`.
- A thread reply has `threadRootId = <root message id>`.
- Thread replies belong to the same `channelId` as the root.
- API must reject replies to missing root messages.
- API must reject replies whose root belongs to a different channel.
- For v0.9, nested threads are not supported: replying to a reply should attach to the original root.

Server/API:

- `GET /api/channels/:id/messages`
  - returns only top-level messages by default.
  - includes `replyCount` and `latestReplyAt` for root messages.
- `GET /api/messages/:id/thread`
  - returns `{ root, replies }`.
- `POST /api/channels/:id/messages`
  - accepts optional `threadRootId`.
  - creates a top-level message when omitted.
  - creates a thread reply when present.
- Browser event:
  - keep `message:new` for top-level messages.
  - add `thread:message:new` for replies, or reuse `message:new` with `threadRootId` and handle it explicitly.
  - prefer explicit `thread:message:new` for clarity.

Shared protocol:

- Extend `Message` in `packages/shared/src/protocol.ts`.
- Extend validation schemas in `packages/shared/src/validation.ts`.
- Update web API types in `packages/web/src/api.ts`.

Database migration:

- Add nullable `thread_root_id` and `mentions` columns to `messages`.
- Existing messages default to top-level.
- Add migration fallback in `packages/server/src/db.ts` similar to previous `ALTER TABLE ... ADD COLUMN` patterns.

Cloudflare hub:

- Mirror the same schema additions and behavior in `packages/cloudflare`.
- Do not let local server and Cloudflare drift.

### 5. Layout

Desktop:

- Main chat remains central.
- Thread panel opens on the right, replacing or sharing space with existing right-side details where needed.
- Width target: 360-440px.
- The panel should have:
  - root message preview
  - reply list
  - compact composer
  - close button

Mobile/narrow:

- Thread panel becomes an overlay or replaces the main chat view.
- No horizontal overflow.

Visual tone:

- Quiet, work-focused, not a marketing UI.
- Use restrained colors and clear spacing.
- Avoid nested cards.
- Keep message rows unframed; use hover affordances instead of heavy boxes.

## Risk Notes

- Message schema touches both local server and Cloudflare; keep tests symmetric.
- Markdown libraries can change CSS defaults; scope styles tightly to message content.
- Mention parsing should not become routing logic in v0.9. Rendering and metadata are enough.
- Thread replies and task conversion need clear behavior: v0.9 should allow creating a task from a root message; creating tasks from replies can be added if cheap, but is not required.
