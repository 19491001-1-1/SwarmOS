# v1.0 — Lightweight Agent Roles & Capabilities

> **给 Claude Code 的执行指令：** 实现本文档描述的 v1.0 窄切面。按 TDD 工作：先写失败测试，再实现，再跑测试确认通过。完成后运行 `pnpm verify`；如果触及 Cloudflare Worker，再运行 wrangler dry-run。

## 背景调整

v1.0 不做复杂的人类组织架构。部门、汇报线、组织图很多是人类组织为了解决人性、责任和管理成本设计出来的机制，不应直接复制给 Agent 系统。

本版本只做轻量角色画像：让系统知道每个 Agent 适合做什么、不能做什么、什么时候应该找它、交接时应该给它什么上下文。

## 目标

把当前 Agent profile 扩展成可用于分工、委派和目录查询的轻量角色/能力模型。

核心问题：

- 这个 Agent 是什么角色？
- 它负责什么？
- 它擅长什么？
- 它不适合做什么？
- 什么情况下应该把工作交给它？
- 它偏好怎样的输入和交付方式？

## 数据模型

在 `Agent.organization` 或后续重命名字段上保持轻量结构。字段名可以沿用当前实现，但产品语义应从“组织架构”转向“角色能力画像”。

```ts
{
  role?: string;
  responsibilities?: string[];
  capabilities?: string[];
  workingStyle?: string;
  handoffPreference?: string;
  constraints?: string[];
  examples?: string[];

  // 兼容已有字段，保留但不作为 v1.0 重点
  department?: string;
  roles?: string[];
  managerId?: string;
  backupAgentIds?: string[];
  availability?: 'available' | 'unavailable' | 'overloaded';
}
```

说明：

- `role` 是用户容易理解的单一主角色，例如 `产品经理`、`工程师`、`研究员`。
- `capabilities` 用于机器匹配，例如 `requirements`, `coding`, `review`, `research`。
- `responsibilities` 描述长期职责。
- `handoffPreference` 描述交接时需要什么输入。
- `constraints` 描述边界，例如不能部署、不能做法务判断。
- `examples` 帮助用户和其他 Agent 理解典型任务。

## 实现范围

- Shared protocol/validation 支持轻量角色能力字段。
- Server SQLite/Cloudflare Durable Object 持久化这些字段。
- `POST /api/agents` 和 `PATCH /api/agents/:id` 可写入角色能力画像。
- Web Agent 详情页能编辑和展示这些字段。
- `crewden agent list` 通过 server info 返回角色能力画像。
- `crewden agent resolve` 支持按 role、capability、responsibility、handoffPreference 匹配。
- delegation/handoff 复用 resolve 行为。

## 不做的事

- 不引入独立 Department/Role 表。
- 不做组织架构图。
- 不做汇报关系流程。
- 不做 manager approval。
- 不做复杂负载调度。
- 不要求用户先建组织，再建 Agent。

## 验收条件

- 用户能在 Agent profile 中设置角色、职责、能力、工作风格、交接偏好、约束和示例。
- `crewden agent list` 能展示这些信息。
- 输入角色或能力关键词时，resolve 能返回对应 Agent 或候选列表。
- 委派时可以根据职责/能力找到合适 Agent，而不是只靠 name/displayName。
- 旧的 `organization` 字段数据兼容，不破坏已有用户数据。
- `pnpm verify` 全绿。
- Cloudflare Worker dry-run 通过。
