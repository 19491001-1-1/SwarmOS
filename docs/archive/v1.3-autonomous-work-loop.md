# v1.3 — Autonomous Work Loop

> **给 Claude Code / Codex 的执行指令：** 实现本文档描述的 v1.3。按 TDD 工作：先写失败测试，再实现，再跑测试确认通过。完成后运行 `pnpm verify`；如果触及 Cloudflare Worker，也必须运行 `pnpm --filter @crewden/cloudflare exec wrangler deploy --dry-run`。

## 背景

当前 Agent 多数时候是被动响应 mention、DM 或 daemon delivery。v1.3 的目标是让 Agent 像团队成员一样稳定“上班”：看到自己的 inbox、认领适合自己的任务、推进、汇报、遇到阻塞时升级。

## 目标

- Agent 启动或 daemon reconnect 后自动检查 open work。
- Agent 能看到统一 inbox：mentions、DM、assigned tasks、claimable tasks、reminders、pending reviews。
- Agent 能认领适合自己 role/capability 的未分配任务。
- 长任务有 heartbeat/progress update。
- 阻塞、缺信息、超时能形成 escalation。

## 非目标

- 不做复杂多 daemon 调度。
- 不做成本优化和 SLA 报表，v1.8 草稿处理。
- 不让 Agent 无限制自动 claim 所有任务。
- 不绕过用户或 reviewer 的确认机制。

## 数据模型

```ts
type WorkItemKind =
  | 'mention'
  | 'dm'
  | 'assigned_task'
  | 'claimable_task'
  | 'reminder'
  | 'review_request'
  | 'blocked_escalation';

type AgentInboxItem = {
  id: string;
  kind: WorkItemKind;
  agentId: string;
  channelId?: string;
  messageId?: string;
  taskId?: string;
  goalId?: string;
  priority: 'low' | 'normal' | 'high' | 'urgent';
  summary: string;
  dueAt?: string;
  createdAt: string;
};

type TaskProgressEvent = {
  id: string;
  taskId: string;
  agentId: string;
  type: 'claimed' | 'started' | 'heartbeat' | 'blocked' | 'handoff' | 'completed';
  detail: string;
  createdAt: string;
};
```

## 实现任务

### Task 1：Unified inbox API

Server 和 Cloudflare 增加：

```http
GET /internal/agent/:agentId/inbox
GET /internal/agent/:agentId/work
```

`inbox` 返回：

- unread deliveries / mentions
- DMs
- assigned open tasks
- tasks whose capability matches this Agent
- reminders
- review requests

要求：

- 结果按 priority + createdAt 排序。
- claimable task 不应无限返回，默认 limit 20。
- 每个 item 必须带可执行 summary。

### Task 2：Task claim & progress API

新增：

```http
POST /internal/agent/:agentId/tasks/:taskId/claim
POST /internal/agent/:agentId/tasks/:taskId/progress
POST /internal/agent/:agentId/tasks/:taskId/block
POST /internal/agent/:agentId/tasks/:taskId/escalate
```

规则：

- 已分配给其他 Agent 的任务不能被 claim，除非显式 handoff。
- claim 时记录 `TaskProgressEvent`。
- block 必须带原因和需要谁提供什么信息。
- escalate 会发 channel/thread message 或创建 visible task update。

### Task 3：Daemon work loop

在 `AgentProcessManager` 或 daemon client 中增加轻量 work loop：

- daemon connected 后，对 autoStart Agent 启动并发送 inbox summary。
- Agent idle 后可按配置检查 inbox。
- 不要在 agent 运行中重复启动导致互相停止。
- 每个 Agent 需要 per-agent queue，不能因为新建/启动另一个 Agent 停掉当前 Agent。

配置建议：

```ts
autoWork?: boolean;
workCheckIntervalMs?: number;
maxClaimableTasksPerRun?: number;
```

默认：

- `autoWork` 跟随 Agent autoStart，但用户可关闭。
- interval 至少 60 秒，避免刷屏。

### Task 4：CLI

新增：

```bash
crewden inbox
crewden work list
crewden task claim <taskId>
crewden task progress <taskId> --detail "..."
crewden task block <taskId> --reason "..." --needs "..."
crewden task escalate <taskId> --reason "..."
```

要求：

- `crewden inbox` 是 Agent 首选入口。
- 输出包含下一步建议，但仍是 JSON。

### Task 5：Bridge prompt

更新 Agent system prompt：

- 启动后先读 `crewden inbox`。
- 有 assigned task 时优先推进。
- claim 任务前确认自己 role/capability 匹配。
- 卡住时 block/escalate，不要编造。
- 长任务要 heartbeat。

### Task 6：Web visibility

Web 最小可见能力：

- Agent detail panel 展示当前 inbox/work summary。
- Task card 展示 claimed by、last heartbeat、blocked reason。
- 阻塞任务在 Task Board 中醒目标记。

## Implementation Notes

- Unified inbox is computed from assigned open tasks, claimable matching tasks, pending reminders, and recent DMs.
- Claim/progress/block/escalate are internal agent APIs and CLI commands; progress events are stored in `TaskContext.progressEvents`.
- Web Task Board displays claimed-by, last progress, and blocked reason/needs. Agent detail shows a compact work summary.
- The daemon prompt now directs agents to start from `crewden inbox`, claim only matching work, and use progress/block/escalate instead of inventing progress.

## 手动验证

1. 创建一个无 assignee 但 capability 为 `coding` 的任务。
2. 启动工程师 Agent。
3. 执行 `crewden inbox`，确认能看到 claimable task。
4. 执行 `crewden task claim <taskId>`。
5. 执行 progress 和 block，确认 Web Task Board 更新。
6. 重连 daemon，确认 autoStart Agent 收到 inbox summary。
