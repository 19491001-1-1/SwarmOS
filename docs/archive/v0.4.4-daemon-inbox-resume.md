# v0.4.4 — Reliable Delivery, Daemon Inbox & Resume

> **给 Claude Code 的执行指令：** 实现本文档描述的所有功能。按 TDD 工作：先写失败测试，再实现，再跑测试确认通过。循环直到所有验收条件满足。完成后运行 `pnpm verify` 确认全绿。

---

## 背景

参考一套成熟的 daemon 设计后，需要先澄清一个设计结论：v0.4.4 的目标不是复刻完整协作 runtime protocol，而是先做可靠投递与恢复调度层。

本版本的第一原则是：**消息不能丢**。

Codex direct busy injection 可以作为 future/best-effort 能力，但不能作为硬验收；任何 direct/notification 投递失败都必须 fallback 到 inbox。

同时，Codex、Claude Code、Gemini 不能按同一种 transport 处理：

- Codex 有 `app-server`，适合长期会话和 busy 时 direct steer，但不在本版本强制实现。
- Claude Code 没有 `app-server`，但支持 `stream-json` stdin/stdout，可做 session resume 和 notification。
- Gemini 没有 `app-server`，当前更适合 MCP 配置 + CLI prompt；不应假设支持 stdin notification。

daemon 的关键能力不只是启动进程，还包括：

- agent 未运行时缓存 `idleConfig`，并能用新消息唤醒
- agent 正在启动时将消息放入 starting inbox
- agent 正在运行但 busy 时，优先保证消息入队，notification/direct 只做 best-effort
- agent 退出后如果 inbox 还有消息，自动续跑下一轮
- 进程退出后不清除 `idleConfig`/sessionId，后续消息仍能唤醒

当前 crewden 的 daemon 还比较简单：收到 `agent:deliver` 后直接 `deliverMessage()`，缺少完整 inbox/resume 策略。这会让 v0.4.2 的 delegation wake 能力在复杂场景下不稳定。

---

## 目标

1. daemon 支持 per-agent inbox，确保消息不丢。
2. 进程退出后保留 `idleConfig` 和 sessionId，后续消息能唤醒 agent。
3. inactive/idle agent 收到消息时能用 wakeMessage 启动。
4. starting agent 收到消息时先排队，启动完成后按 FIFO 处理。
5. working agent 收到消息时优先保证入队；notification/direct 都必须可失败回退。
6. wake prompt 包含 unread summary，提示 agent 有多少条排队消息以及如何读取。
7. 引入 runtime transport/capabilities，不再假设所有 runtime 支持 stdin。
8. Claude Code 优先升级到 `stream-json` transport，保存 sessionId 并 best-effort `--resume`。
9. Gemini 保持 conservative：写入 `.gemini/settings.json` MCP 配置，busy 消息进入 inbox。
10. 为 v0.4.5 memory 预留 prompt/context 接口，但不实现 `MEMORY.md`。
11. 与 v0.4.1 的 auto-start/recovery、v0.4.3 的 agent-facing CLI 兼容。

## Transport 设计

新增明确的 transport/capabilities 分层：

```ts
type RuntimeTransport = 'oneshot' | 'stream-json' | 'app-server' | 'mcp';

type BusyDeliveryMode = 'best-effort-direct' | 'notification' | 'inbox';

type RuntimeCapabilities = {
  transport: RuntimeTransport;
  supportsStdinDelivery: boolean;
  busyDeliveryMode: BusyDeliveryMode;
  supportsSessionResume: boolean;
  supportsMcpBridge: boolean;
};
```

v0.4.4 目标矩阵：

| Runtime | Transport | Busy Delivery | Resume | MCP Bridge |
|---------|-----------|---------------|--------|------------|
| Claude Code | `stream-json` | `notification` | yes, best-effort | optional/deprecated shim |
| Codex | `oneshot` for now | `inbox` for now | no | no |
| Gemini | `mcp` + CLI prompt | `inbox` | optional only if CLI emits session | yes |

注意：

- Codex `app-server` 是未来增强，不在本版本强制实现。
- direct busy injection 只允许作为 best-effort；失败必须进入 inbox。
- Gemini 不支持 stdin notification，busy 时必须排队。
- Claude 的 notification 不是直接把业务消息塞给它处理，而是提示它通过 `crewden message check/read` 拉取。

## 状态模型

daemon 内部维护：

```ts
type LocalAgentRuntimeState = {
  agentId: string;
  registered: boolean;
  idleConfig?: AgentRuntimeConfig;
  processStatus: 'starting' | 'active' | 'idle' | 'exited' | 'error';
  activeConfig?: AgentRuntimeConfig;
  channelId?: string;
  sessionId?: string;
  inbox: AgentDelivery[];
  startedAt?: string;
  lastActiveAt?: string;
};
```

语义要求：

- `registered=true` 表示 daemon 本地已知这个 agent 的 config，后续可被唤醒。
- `idleConfig` 是进程退出后的恢复配置；只要 agent 仍注册，就不能因为进程退出而清除。
- `processStatus` 只描述本地进程状态，不承担“是否可唤醒”的含义。
- `sessionId` 是 best-effort resume 信息；丢失不能导致消息丢失。
- `inbox` 是唯一硬保证路径；任何非 inbox 投递失败都必须重新入队。

runtime driver 上下文增加 session、formatted message、unread summary 和 context block：

```ts
type AgentSpawnContext = {
  agentId: string;
  config: AgentRuntimeConfig;
  workspaceDir: string;
  transcriptFile: string;
  userMessage: string;
  formattedMessage: string;
  sessionId?: string;
  unreadSummary?: {
    queuedCount: number;
    newestMessageAt?: string;
  };
  contextBlocks?: PromptContextBlock[];
};

type PromptContextBlock = {
  id: string;
  title: string;
  content: string;
};
```

`contextBlocks` 为 v0.4.5 memory/notes 预留。v0.4.4 不读取 `MEMORY.md`，但 prompt/formatter 结构不能阻碍后续插入 memory block。

## 实现任务

### Task 1：Driver transport/capabilities 声明

- `packages/daemon/src/drivers/types.ts`
  - 新增 `RuntimeCapabilities`
  - 新增 `RuntimeTransport`
  - driver export capabilities
- Claude Code：
  - `transport = 'stream-json'`
  - `supportsStdinDelivery = true`
  - `busyDeliveryMode = 'notification'`
  - `supportsSessionResume = true`
- Codex：
  - v0.4.4 先保持当前 CLI 模式
  - `busyDeliveryMode = 'inbox'`
  - 不在本版本实现 app-server
- Gemini：
  - `transport = 'mcp'`
  - `supportsStdinDelivery = false`
  - `busyDeliveryMode = 'inbox'`
  - `supportsMcpBridge = true`

测试：

- driver capabilities 快照测试覆盖三种 runtime。
- 不允许 Gemini capabilities 声称支持 stdin delivery。
- 不允许任何 driver 把 direct delivery 作为无 fallback 的硬依赖。

### Task 2：Claude Code stream-json transport

把 Claude driver 从 text one-shot 升级为 stream-json：

```bash
claude \
  --dangerously-skip-permissions \
  --verbose \
  --output-format stream-json \
  --input-format stream-json \
  --append-system-prompt "<standingPrompt>" \
  --model "<model>"
```

如果存在 `sessionId`：

```bash
claude ... --resume "<sessionId>"
```

启动后 daemon 往 stdin 写入当前 wake prompt 或 formatted message：

```json
{
  "type": "user",
  "message": {
    "role": "user",
    "content": [{ "type": "text", "text": "<wakePromptOrFormattedMessage>" }]
  },
  "session_id": "<optional>"
}
```

解析 Claude stdout：

- `system/init` + `session_id` -> `session_init`
- `assistant/message/content/text` -> `message`
- `assistant/message/content/tool_use` -> `activity`
- `result` -> `turn_end`
- 非 JSON 日志忽略或记录到 activity

测试：

- build command 包含 `--input-format stream-json` 和 `--output-format stream-json`。
- 有 sessionId 时包含 `--resume`。
- `encodeStdinMessage()` 输出合法 Claude stream-json。
- `parseOutput()` 能解析 text、session init、turn end。
- stream-json 写 stdin 失败时，消息重新进入 inbox。

### Task 3：Gemini MCP 配置与 conservative inbox

Gemini 保持 CLI prompt 启动，但要补齐协作型 MCP 配置基础：

- 启动前写入 `workspaceDir/.gemini/settings.json`
- 配置 chat MCP server，优先指向当前 repo 的 daemon chat bridge
- 如果 MCP bridge 尚未完整实现，可先写入 v0.4.6 将复用的配置结构，并用测试锁定文件格式
- Gemini busy 时所有新消息进入 inbox，不往 stdin 写

配置形状：

```json
{
  "mcpServers": {
    "chat": {
      "command": "node",
      "args": ["<bridgePath>", "--agent-id", "<agentId>", "--server-url", "<serverUrl>", "--auth-token", "<token>"]
    }
  }
}
```

测试：

- start Gemini agent 会写 `.gemini/settings.json`。
- Gemini capabilities 为 inbox。
- Gemini working 时 deliver 第二条消息只进入 inbox，不写 stdin。

### Task 4：AgentProcessManager inbox

- `startAgent(agentId, config, channelId, wakeMessage?)`
  - 写入或更新 `idleConfig`
  - 设置 `registered = true`
  - 如果 agent 已 starting/active，不重复杀进程
  - 如果提供 wakeMessage，将其加入 inbox
  - 如果存在 sessionId 且 runtime 支持 resume，传给 driver
- `deliverMessage(agentId, delivery, config?)`
  - 先把消息 append 到 inbox
  - agent 不存在但有 config：注册 `idleConfig` 并启动
  - agent 不存在且没有 config/idleConfig：返回明确错误并上报 activity error
  - starting：只入队
  - idle/exited：启动下一轮处理消息
  - active/working：
    - notification：best-effort 注入“有新消息，请 check_messages”
    - direct：仅 future/best-effort；失败后消息仍留在 inbox
    - inbox：默认可靠路径，等待当前进程结束
  - process exit 后如果 inbox 非空，按 FIFO 自动启动下一轮
  - process exit 后如果 inbox 为空，保留 `idleConfig`、`sessionId`，`processStatus = 'idle' | 'exited'`

测试：

- deliver 到不存在但带 config 的 agent 会 start。
- startAgent 会持久化 `idleConfig`，进程退出后仍可被唤醒。
- starting 期间多条 deliver 进入 inbox。
- working + inbox runtime 会排队。
- notification/direct 投递失败后消息仍留在 inbox。
- process exit 后 inbox 非空会继续启动下一轮。
- 没有 config 且本地无 idle config 时返回明确错误，不吞消息。

### Task 5：Session 追踪

- daemon 处理 driver 输出：
  - `session_init` 保存 sessionId
  - 发送 `agent:session` 给 hub
- local state 保留 sessionId
- agent 正常退出时不清除 `idleConfig`/sessionId
- 只有 `supportsSessionResume` 的 runtime 才把 sessionId 传回 driver
- resume 是 best-effort：`--resume` 失败时允许冷启动，但 inbox 消息不能丢

测试：

- Claude session init 被保存。
- 下一轮 Claude 启动使用 `--resume`。
- Claude resume 失败时能冷启动并继续处理 inbox。
- Gemini/Codex 不因缺少 sessionId 失败。

### Task 6：Wake prompt 与消息格式化

实现统一 formatter：

```text
[target=#general msg=<id> time=<iso> type=human] @sender: content
[target=dm:<fromAgentId> msg=<id> time=<iso> type=agent] @sender: content
```

用于：

- wake prompt
- Claude stream-json stdin delivery
- notification delivery
- inbox 下一轮启动 prompt

wake prompt 必须包含 unread summary：

```text
You have <N> queued messages.
Use `crewden message check` or `crewden message read --channel <channel> --limit <N>` if needed.
```

如果当前启动只消费 inbox 里的第一条消息，也要告诉 agent 还有多少条排队消息。不要在 v0.4.4 实现完整 history/search，读取仍交给 v0.4.3 CLI。

prompt builder 要预留 context block 插槽。v0.4.4 可以只传空数组；v0.4.5 再插入 memory/notes。

### Task 7：Server/Hub ack 与状态

- daemon 收到 `agent:deliver` 后立即 `agent:deliver:ack`
- 成功排队也算 ack
- 处理失败时发送 `agent:activity type=error`
- server 不因 ack 就认为任务完成，只代表 daemon 接收
- `agent:session` 可被 server/cloudflare 接收并保存或安全忽略；不能破坏现有状态机
