---

## 背景

v0.4.3 会提供 agent-facing CLI。参考一套成熟的 daemon 设计，部分 runtime 还可以通过 MCP tools 调用平台能力。MCP 相比 CLI 的优势：

- 工具 schema 明确
- 调用返回结构化结果
- 适合 Claude/Codex 等支持 MCP 或 app-server tool bridge 的 runtime
- 比 stdout marker 更不容易格式错误

本版本目标是补齐“agent 调用协作工具”的第二条通道：CLI 为主，MCP bridge 为辅。

---

## 目标

1. daemon 提供本地 MCP stdio server
2. MCP tools 调用同一套 internal agent API
3. runtime driver 能把 MCP 配置传给支持的 agent CLI
4. MCP 与 CLI 能力保持一致，不引入第二套业务逻辑
5. stdout marker 继续作为 fallback

## 实现任务

### Task 1：MCP bridge package

新增：

```text
packages/daemon/src/mcp/
```

实现：

- stdio MCP server
- 读取环境变量：
  - `CREWDEN_AGENT_ID`
  - `CREWDEN_SERVER_URL`
  - `CREWDEN_AGENT_TOKEN_FILE`
- 所有 tool 转发到 internal agent API
- 返回结构化 JSON

### Task 2：Tool schema

为每个 tool 定义输入 schema：

```ts
send_message({ channelId, content })
send_dm({ to, content })
delegate_agent({ to, content, startIfInactive })
read_history({ channelId, limit })
```

测试无效入参应返回 tool error，而不是进程崩溃。

### Task 3：Driver 集成

- Claude：
  - 生成 MCP config
  - 启动参数引用该 MCP config
- Codex：
  - 如当前 app-server 支持 MCP，则传入配置
  - 如不支持，保持 CLI-only
- Gemini：
  - 保持 CLI-only，除非已有稳定 MCP 配置方式

### Task 4：Prompt 更新

standing prompt 明确优先级：

1. MCP tools（如果可用）
2. `crewden` CLI
3. stdout marker fallback

并说明：

- 复杂查询优先使用 structured tools
- token 不可打印
- 写作输出不要伪造工具调用结果

### Task 5：兼容性测试

- MCP tool 调用 internal API 成功
- agent token 错误时 tool 返回 unauthorized
- CLI 与 MCP 发送消息最终写入同一张 messages 表
- delegate_agent tool 复用 v0.4.2 delegation 逻辑
