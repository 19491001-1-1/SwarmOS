# v1.7 Agent Planning & Verification Roadmap

## 版本目标

让 agent 在执行前能形成计划，执行后能自我验证；让 goal decomposition 从规则驱动升级为 LLM-assisted，但保留可审计、可测试、可降级的结构化输出。

## 非目标

- 不做全自动高风险执行。
- 不把所有任务理解交给 LLM；规则仍保留为 fallback。
- 不实现完整 DAG 编排。

## 为什么现在做

v1.6 解决可靠投递和状态约束后，下一瓶颈是“agent 盲干”和“说 done 但没验证”。v1.7 补 plan-before-execute 与 self-verification，直接提升 Coding Agent 的问题解决闭环。

## 核心用户故事

1. 作为用户，我发一个较大 coding goal，系统能给出结构化计划、风险和建议 agent。
2. 作为 coding agent，我收到 task 后先输出 plan，系统记录并可要求确认。
3. 作为 coding agent，我完成实现前必须运行验证命令，失败则不能标 done。
4. 作为 agent，我遇到缺上下文/权限/测试环境失败时主动 block 并说明下一步。

## 数据模型/API/协议建议

- 新 marker：`[[CREWDEN_PLAN]] {"taskId":"...","steps":[...],"risks":[...],"verification":[...]}`。
- 新 marker：`[[CREWDEN_VERIFY_RESULT]] {"taskId":"...","command":"pnpm test","status":"passed|failed","summary":"..."}`。
- Task context 增加 `executionPlan`、`verificationResults`、`requiresPlanApproval`。
- Goal planner 输出 schema：`objective, assumptions, steps[], risks[], clarificationQuestions[], suggestedAgents[], acceptanceCriteria[]`。

### Task 4：Plan-before-execute gate

- Failing test：`pauses high risk task until plan is approved` in `agentProcessManager.test.ts`。
- 命令：`pnpm --filter @crewden/daemon test -- agentProcessManager.test.ts -t "plan is approved"`
- 期望失败：agent 直接执行。

- 通过标准：高风险 task 无 approval 不进入 working；approval 后继续。
- Commit：`feat(agent): require plans for high risk tasks`

### Task 5：Self-verification 结果写回

- Failing test：`does not mark task done when verification failed`。
- 命令：`pnpm --filter @crewden/daemon test -- agentProcessManager.test.ts -t "verification failed"`

- 最小实现：`[[CREWDEN_VERIFY_RESULT]]` 写入 task context；done 前检查最新 verification status；失败则 block 或 review_requested。
- 通过标准：验证失败时 task 不会 done，频道收到失败摘要。
- Commit：`feat(agent): require verification before done`

### Task 6：Proactive blocking CLI

- Failing test：`agent cli blocks task with reason and next step` in `agentCli.test.ts`。
- 命令：`pnpm --filter @crewden/daemon test -- agentCli.test.ts -t "blocks task"`

- 通过标准：block 事件、audit、task context 同步更新。
- Commit：`feat(cli): support proactive task blocking`

## 测试用例清单

- Happy：LLM planner 产出 steps；agent plan 被记录；verify passed 后 done。
- Edge：planner 输出多余字段；plan marker 分块输出；verification 命令为空。
- Failure：planner timeout；invalid JSON marker；verify failed；高风险 task 未审批。
- Concurrency/restart：plan approval 前 server restart；多个 verify result 取最新；同 task 多 agent 同时提交 plan。

## 验收标准

- 计划、验证、block 都进入 audit log。
- 高风险任务默认 human-in-the-loop；低风险可配置自动执行。
- planner 可完全关闭并回退规则逻辑。
- Coding Agent 能从 task prompt 看到明确“先 plan、再执行、再 verify”的协议。

## 风险与降级

- LLM planner 不稳定：zod 校验 + rule fallback。
- plan gate 降低速度：只对 high risk 默认启用。
- verification 命令误判：允许 agent 提供 evidence，但必须标明未运行原因。

## 版本依赖

依赖 v1.6 的状态机、audit 和 retry；为 v1.9 human checkpoint 和 DAG 编排提供 plan/verify 原语。
