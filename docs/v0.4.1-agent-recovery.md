# v0.4.1 — Agent Recovery & Daemon Auto Start

> **给 Claude Code 的执行指令：** 实现本文档描述的所有功能。按 TDD 工作：先写失败测试，再实现，再跑测试确认通过。循环直到所有验收条件满足。完成后运行 `pnpm verify` 确认全绿。

---

## 背景

当前 hub/server 在启动或 Cloudflare Durable Object 冷启动时会执行 volatile 状态重置：

- 所有 machine 被标记为 `offline`
- 所有 `starting` / `running` / `working` / `idle` agent 被标记为 `inactive`

这会导致一个明显问题：用户创建或刷新 agent 时，如果触发 hub 冷启动、server 重启或 daemon 重连，其他原本可用的 agent 会在 UI 上表现为“被停止”。同时 daemon `ready` 消息当前上报 `runningAgents: []`，hub 也没有根据 agent 配置和 machine 能力自动恢复 agent。

v0.4.1 的目标是把 agent 生命周期从“全局启动时清空”改为“按 machine 连接状态精准恢复”，并让 daemon 连上后自动启动应该运行的 agent。用户仍然可以手动停止 agent，手动停止后不应被自动拉起。

---

## 目标

1. **不再因为 hub/server 冷启动误停所有 agent**
2. **daemon 连接成功后，自动启动绑定到该 machine 且应该运行的 agent**
3. **保留用户手动停止语义**：用户点 Stop 后，该 agent 不会被 daemon 自动启动
4. **支持 daemon 重启后的恢复**：daemon 重新连接时，hub 能根据持久化状态重新下发 `agent:start`
5. **本地 server 和 Cloudflare hub 行为一致**

---

## 状态模型调整

### Agent 新增字段

建议新增持久化字段：

```ts
type Agent = {
  autoStart: boolean;       // 用户是否希望该 agent 随 daemon 连接自动启动
  lastStopReason?: string;  // manual | machine_offline | daemon_restart | error
  desiredStatus?: string;   // inactive | running
};
```

最小实现可以只落地 `autoStart`：

- 创建 agent 默认 `autoStart = false`
- 用户点击 Start 后设置 `autoStart = true`
- 用户点击 Stop 后设置 `autoStart = false`
- daemon 断开或 hub 冷启动不修改 `autoStart`
- daemon 连接时自动启动 `autoStart = true` 且可由当前 machine 运行的 agent

如果选择实现 `desiredStatus`，则语义为：

- `desiredStatus = 'running'`：用户期望 agent 保持运行，daemon ready 后恢复
- `desiredStatus = 'inactive'`：用户手动停止，不自动恢复
- `status` 仍表示当前观测状态，用于 UI 展示

## 协议与恢复语义

### Daemon → Hub `ready`

保留现有字段：

```ts
{
  type: 'ready',
  machineId: string,
  runtimes: RuntimeId[],
  runningAgents: string[]
}
```

实现要求：

- daemon 应上报当前进程内已注册的 agent id，而不是固定 `[]`
- daemon 刚启动时可以是 `[]`，hub 会负责根据持久化的 `autoStart` / `desiredStatus` 下发启动
- hub 收到 ready 后只对该 machine 做 reconciliation，不做全局 agent 清理

### Hub → Daemon `agent:start`

`agent:start` 保持现有协议，增加幂等要求：

- 如果 daemon 已经注册该 agent，重复 `agent:start` 不应清空 transcript 或破坏当前状态
- 如果 agent 正在处理消息，重复 start 应只确认状态，不应杀掉或重启当前子进程

## 实现任务

### Task 1：移除全局误停逻辑

- `packages/server/src/db.ts`
  - 不再在 `initDb()` 中无条件把所有 active agent 置为 `inactive`
  - machine 可以在进程启动时先标记为 `offline`，但不能顺带停止所有 agent
- `packages/cloudflare/src/index.ts`
  - Durable Object constructor 中不再把所有 active agent 置为 `inactive`
  - 冷启动只恢复 schema，不改变 agent 的用户期望状态
- 将“实际不可达”的状态调整移动到 machine disconnect / daemon ready reconciliation 阶段

### Task 2：持久化 auto-start 意图

- `packages/shared/src/protocol.ts`
  - Agent 类型新增 `autoStart?: boolean` 或 `desiredStatus?: 'inactive' | 'running'`
- `packages/shared/src/validation.ts`
  - PATCH agent schema 支持更新该字段
- `packages/server/src/db.ts`
  - agents 表新增字段，旧数据默认 `false`
- `packages/cloudflare/src/index.ts`
  - Durable Object SQL schema 同步新增字段，旧数据默认 `false`
- `packages/web/src/api.ts`
  - Agent 类型同步新增字段

### Task 3：Start / Stop 语义改造

- `POST /api/agents/:id/start`
  - 成功向 daemon 发送 `agent:start` 后，将 `autoStart` 设为 `true`
  - 当前状态设为 `starting`
- `POST /api/agents/:id/stop`
  - 向 daemon 发送 `agent:stop`
  - 将 `autoStart` 设为 `false`
  - 当前状态设为 `inactive`
- agent create 默认：
  - `status = 'inactive'`
  - `autoStart = false`

### Task 4：Daemon ready 自动恢复

在 server 和 Cloudflare hub 的 `ready` 处理逻辑中实现：

1. upsert 当前 machine，标记 online
2. 合并重复 machine id
3. 找出满足以下条件的 agent：
   - `agent.autoStart === true` 或 `agent.desiredStatus === 'running'`
   - agent 绑定到当前 machine，或未绑定但当前 machine 支持其 runtime
   - 当前 machine 的 `runtimes` 包含 agent runtime
4. 对未出现在 `ready.runningAgents` 中的 agent 下发 `agent:start`
5. 对已经在 `ready.runningAgents` 中的 agent，只更新状态为 `running` 或 `idle`
6. 不处理其他 machine 上的 agent

### Task 5：Machine disconnect 精准降级

daemon WebSocket 断开时：

- 当前 machine 标记为 `offline`
- 只将绑定到该 machine 且当前状态为 `starting/running/working/idle` 的 agent 状态降级为 `inactive` 或 `disconnected`
- 不修改这些 agent 的 `autoStart`
- 下一次同一 machine ready 后，这些 agent 应被自动启动

如果新增 `disconnected` 状态：

- UI 使用灰色或黄色提示，不显示为用户手动停止
- Composer 不应把 disconnected agent 当作可投递对象

### Task 6：Daemon 侧幂等与 runningAgents

- `packages/daemon/src/agentProcessManager.ts`
  - 新增 `listRunningAgentIds()`，返回已注册 agent id
  - `startAgent()` 对同一个 agent id 幂等：已存在时更新 config/channelId，保持 workspace 和 transcript
- `packages/daemon/src/daemonClient.ts`
  - `sendReady()` 使用 `processManager.listRunningAgentIds()`
  - 收到重复 `agent:start` 时不会导致其他 agent 停止

### Task 7：Web UI 表达用户意图

- Agent card / profile 中可显示 auto-start 状态：
  - `AUTO START: ON/OFF`
  - Start 后 ON，Stop 后 OFF
- 不需要新增独立开关；v0.4.1 中先用 Start/Stop 表达用户意图
- 如果实现 `disconnected` 状态，按钮文案保持：
  - disconnected/error/inactive 显示 START
  - starting/running/working/idle 显示 STOP

## 不做的事

- 不实现跨 machine 迁移策略；本版本只恢复到可用且匹配 runtime 的 machine
- 不实现 agent 运行中子进程快照恢复；恢复的是 agent 注册状态和后续消息处理能力
- 不改变现有 CLI runtime 的消息执行模型
- 不删除远端任务分支
