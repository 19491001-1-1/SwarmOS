# v1.6 Reliable Task Flow Roadmap

## 版本目标

让 Crewden 的任务传递链路具备生产可靠性：状态转移可控、失败可恢复、关键动作可审计、重启后不丢状态，并支持最小 dependency graph。

## 非目标

- 不做复杂 DAG UI。
- 不引入 LLM planning。
- 不重写存储层；先在现有 server/db/schema 结构上增量实现。

## 为什么现在做

当前 Crewden 已有 task、delegation、handoff、daemon delivery，但状态转移偏松、失败后停住、部分状态在内存里，3 个以上 agent 协作时缺依赖关系。v1.6 是后续自主规划和编排的地基。

## 核心用户故事

1. 作为 coding agent，我领取 task 后只能按合法状态推进，避免误把 blocked task 标成 done。
2. 作为 daemon，我的 agent 因网络/rate limit 失败时会自动重试，认证失败则升级给人类。
3. 作为项目 owner，我能追溯某个 task 的 handoff、delegation、retry、block、done 全链路。
4. 作为 coding agent，我能声明 task B blocked by task A，系统不应提前执行 B。

## 数据模型/API/协议建议

- Task 增加 `version` 或使用 `updatedAt` 作为 optimistic lock。
- Task 增加 `blockedByTaskIds: string[]` 或单独 `task_dependencies(task_id, blocks_task_id)`。
- 新增 `audit_log`：`id, actorType, actorId, action, entityType, entityId, taskId, agentId, detailJson, createdAt`。
- Daemon retry 元数据：`attempt, maxAttempts, nextRetryAt, lastErrorClass`。
- API：`PATCH /tasks/:id/status` 要求 `expectedVersion`，非法 transition 返回 409/422。

## TDD 任务拆解

### Task 1：定义 Task 状态机

- Failing test：在 `packages/server/test/tasksApi.test.ts` 增加 `rejects invalid task status transition`。
- 命令：`pnpm --filter @crewden/server test -- tasksApi.test.ts -t "rejects invalid task status transition"`
- 期望失败：当前 API 允许直接 patch 到非法状态或没有明确 422。
- 最小实现：在 `packages/server/src/routes/tasks.ts` 增加 `assertTaskTransition(from,to)`；允许 `open -> in_progress -> review -> done`、`open/in_progress -> blocked`、`blocked -> open`、`* -> cancelled`。
- 通过标准：非法 transition 返回 422，合法 transition 仍通过。
- Commit：`feat(tasks): enforce task status transitions`

### Task 2：加入 optimistic lock

- Failing test：`returns conflict when task version is stale`。
- 命令：`pnpm --filter @crewden/server test -- tasksApi.test.ts -t "version is stale"`
- 期望失败：两个并发 patch 都成功。
- 最小实现：task 记录增加 `version`；patch 时比对 `expectedVersion`；成功后 `version + 1`。
- 通过标准：旧 version 返回 409；新 version 成功。
- Commit：`feat(tasks): add optimistic locking`

### Task 3：审计日志最小表与写入

- Failing test：`writes audit log for task status change and handoff`。
- 命令：`pnpm --filter @crewden/server test -- db.test.ts -t "audit log" && pnpm --filter @crewden/server test -- tasksApi.test.ts -t "audit log"`
- 期望失败：无 audit_log 或无记录。
- 最小实现：`schema.ts` 增表；`db.ts` 增 `appendAuditLog/listAuditLogs`；`tasks.ts/internalAgent.ts/delegation.ts` 在状态变化、handoff、delegation 时写入。
- 通过标准：日志包含 actor、action、entity、detailJson，不包含 secret。
- Commit：`feat(audit): record task flow events`

### Task 4：daemon 自动重试与 escalation

- Failing test：`retries transient agent failure before marking error`、`blocks task on permanent failure` in `packages/daemon/test/agentProcessManager.test.ts`。
- 命令：`pnpm --filter @crewden/daemon test -- agentProcessManager.test.ts -t "retries transient"`
- 期望失败：当前 close handler 直接 error/idle。
- 最小实现：在 `agentProcessManager.ts` close handler 分类 stderr：`ENOTFOUND|ECONNRESET|rate limit|timeout` 为 transient；`auth|permission denied|command not found` 为 permanent；transient 最多 2 次指数退避；permanent 调 server CLI/API block task 并发 activity error。
- 通过标准：fake driver 失败两次后第三次成功；永久失败不会重试且产生 block/escalation 事件。
- Commit：`feat(daemon): retry transient agent failures`

### Task 5：daemon registry 状态持久化

- Failing test：`restores known daemon machines after server restart` in `daemonSocket.test.ts`。
- 命令：`pnpm --filter @crewden/server test -- daemonSocket.test.ts -t "restores known daemon"`
- 期望失败：重启后 registry 为空。
- 最小实现：machine/daemon connection 的稳定元数据写 DB；启动时恢复 machine list，connection live 状态仍由 websocket 决定。
- 通过标准：重启后可展示 known machine，但 offline 状态准确。
- Commit：`feat(daemon): persist daemon registry metadata`

### Task 6：最小 Task Dependency Graph

- Failing test：`does not deliver task while dependency is open`、`unblocks dependent task when blocker is done`。
- 命令：`pnpm --filter @crewden/server test -- tasksApi.test.ts -t "dependency"`
- 期望失败：依赖不存在或不会影响 delivery。
- 最小实现：`task_dependencies` 或 `blockedByTaskIds`；创建/更新时检测循环；`notifyTaskAssignee` 前检查依赖状态；blocker done 后触发 dependent delivery。
- 通过标准：循环依赖 422；blocked task 不投递；blocker done 后自动投递。
- Commit：`feat(tasks): add minimal dependency graph`

## 测试用例清单

- Happy path：合法状态转移；transient 失败后成功；依赖完成后投递。
- Edge：重复 patch 同一 version；空 dependency；self-dependency。
- Failure：非法状态；永久错误；audit 写入失败时主流程如何降级。
- Concurrency/restart：并发 patch；server restart 后 registry 和 audit 可查；daemon retry 过程中重启不重复执行超限。

## 验收标准

- 所有新增行为都有 failing test 记录。
- `pnpm --filter @crewden/server test`、`pnpm --filter @crewden/daemon test` 通过。
- 任何 task flow 关键动作都能在 audit log 查到。
- 失败分类和 retry 次数可配置，有安全默认值。

## 风险与降级

- 状态机过严导致旧客户端失败：先允许 legacy patch 但记录 warning，下一小版本收紧。
- retry 误判永久错误：保守重试 1 次并 audit。
- dependency graph 复杂度上升：v1.6 只做 blocked-by，不做条件分支。

## 版本依赖

v1.6 是 v1.7/v1.8/v1.9 的基础。v1.7 的 plan-execute-verify 依赖状态机；v1.9 DAG 依赖 dependency graph。
