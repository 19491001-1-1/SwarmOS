# v0.9 — 聊天体验、状态感知与 Thread

> 本文档是可直接交给 Claude Code / Codex 执行的版本计划。执行时必须从 `main` 新建分支，按 TDD 推进，持续运行测试，直到验收清单全部通过。

## 目标

把当前群聊从“工程调试日志”升级成“成熟团队协作聊天界面”。

v0.9 的核心交付：

1. 头像上清晰展示每个人/Agent 的在线、空闲、工作中、报错等状态。
2. 群聊消息以中文优先、可阅读的 Markdown 方式展示。
3. `@某人` 能被识别并渲染成易读的 mention。
4. 支持 Thread，让某个话题可以在侧边栏深入讨论，不污染主频道。
5. 修正当前截图里暴露出的真实体验问题：裸露 `\n`、Markdown 表格未渲染、文本行过长、黑框过重、右侧面板挤压聊天区。

## 基于当前截图的观察

从实际页面看，当前主要问题不是缺一个小功能，而是聊天阅读体验整体还比较粗糙：

- 消息内容仍是纯文本，`**标题**`、Markdown 表格、列表、链接都没有正常渲染。
- 部分 Agent 输出里出现字面量 `\n\n`，说明服务端或渲染层没有把转义换行整理成真实换行。
- 长段中文和 URL 在一行里横向铺开，阅读压力大。
- 表格以源码形式展示，中文表格非常难读。
- 消息 hover action 的 `-> TASK` 按钮常驻在右侧，视觉噪音偏大。
- 当前像素风黑框过重，适合作为早期原型，但不适合承载长内容阅读。
- 左侧黄色、右侧 Agent 面板、黑色边框共同占用大量视觉注意力，主聊天内容反而不够突出。
- 右侧 Agent 面板宽且信息重复，桌面大屏还可以，窄屏时会挤压聊天正文。
- Agent 状态已经有文字，但用户需要一眼从头像或列表看到谁正在工作。
- Thread 尚未存在，导致“转机建议”“待办交接”“北京穿衣建议”这类话题都堆在主频道里。

v0.9 的设计目标就是针对这些真实问题做一次体验升级。

## 非目标

- 不重做整个产品信息架构。
- 不做富文本编辑器，输入框仍然是纯文本，支持 Markdown 即可。
- 不给 DM 私聊加 Thread。
- 不做跨频道 Thread 搜索。
- 不做完整 Slack 级 unread/thread inbox，v0.9 只做当前频道内的 Thread 体验。
- 不把 mention 解析直接升级成复杂路由系统。mention 先用于展示与后续扩展的结构化数据。

## 设计原则

### 中文优先

- UI 文案、空状态、按钮、状态文案默认使用中文。
- 技术字段可以保留英文，例如 runtime、commit、hub version。
- Agent 输出中中英文混排时，中文阅读优先：行高、字重、段落间距要适合中文。

建议文案：

| 当前 | 建议 |
|---|---|
| `WORKSPACE` | `工作区` |
| `SEARCH` | `搜索` |
| `TASKS` | `任务` |
| `CHANNELS` | `频道` |
| `AGENTS` | `成员 / Agent` |
| `MACHINES` | `机器` |
| `Message #general` | `发消息到 #general` |
| `SEND` | `发送` |
| `-> TASK` | `转任务` |
| `STATUS: RUNNING` | `在线` / `工作中` / `空闲` |
| `AUTO START: ON` | `自动启动已开启` |

### 阅读优先

- 主聊天区是第一优先级，左右侧栏都不能抢视觉重心。
- 消息不能用过重边框包裹。
- 长文内容要靠 Markdown、段落、表格、代码块提升可读性。
- Hover action 默认隐藏，鼠标悬停时出现。
- 群聊消息列表应该像 Slack 一样密集但不拥挤。

### 状态一眼可见

- 用户不应该读 `STATUS: RUNNING` 才知道状态。
- 头像状态点和 Agent 列表状态点必须统一。
- “正在工作/思考”要比普通在线更显眼，但不能像告警一样刺眼。

### Thread 不打断主频道

- Thread 是主频道的侧边聚焦，不是新频道。
- 主频道只展示根消息和回复摘要。
- Thread 面板展示根消息、回复列表和独立输入框。

## 当前基线

- `messages` 表目前只有 `id`, `channelId`, `senderName`, `content`, `agentId`, `createdAt`。
- `ChannelView` 用 `whiteSpace: pre-wrap` 直接显示内容。
- `Composer` 有 Agent 选择器，但 typed mention 没有解析。
- Agent 已有状态：`inactive | starting | running | working | idle | error`。
- Activity 已有 `thinking | working | output | idle | sending | error`。
- WebSocket 已能推送消息、Agent、任务等更新。
- 没有 Thread 数据模型。

## 产品需求

### 1. 头像与状态

每个可见头像右下角展示状态点。Agent 列表中也复用同一套状态语言。

状态映射：

| 状态来源 | 展示 | Tooltip |
|---|---|---|
| `working` | 蓝紫色呼吸点或细环动画 | 工作中 |
| 最新 activity 为 `thinking` | 蓝色呼吸点 | 思考中 |
| `starting` | 小型旋转环 | 启动中 |
| `running` | 绿色实心点 | 在线 |
| `idle` | 灰绿色小点 | 空闲 |
| `error` | 红色点 | 异常 |
| `inactive` / 缺失 | 灰色空心点 | 离线 |
| human user | 绿色点 | 你 |

要求：

- 状态点不能改变头像尺寸。
- 分组消息不重复头像，也不重复状态点。
- 状态变化不造成消息列表抖动。
- Agent 列表中的状态文案改成中文短标签：`工作中`、`空闲`、`在线`、`异常`。

建议实现：

- 新增 `PresenceAvatar` 组件。
- 输入：`name`, `isAgent`, `status`, `latestActivity`。
- 由 `App.tsx` 根据 `agents` 和 `activitiesByAgent` 计算 `agentPresenceById`。

### 2. 群聊消息 Markdown 渲染

消息必须安全渲染 Markdown，至少支持：

- 段落
- 加粗、斜体、行内代码
- 代码块
- 有序/无序列表
- 引用块
- 表格
- 链接

强制要求：

- 不允许渲染原始 HTML。
- 链接新窗口打开，带 `rel="noreferrer"`。
- 代码块可横向滚动，不能撑破页面。
- 表格必须渲染成真正表格，不再显示 Markdown 源码。
- 中文长段落需要合理行宽和行高。
- 内容里出现字面量 `\n` / `\n\n` 时，要在显示前规范化为真实换行。

推荐依赖：

```bash
pnpm --filter @mini-slock/web add react-markdown remark-gfm
```

建议组件：

- `MessageContent.tsx`
- `normalizeMessageContent(content: string): string`
- `renderMentionAwareMarkdown(...)`

验收样例：

```md
**穿衣建议**

| 日期 | 外套 | 内搭 |
|---|---|---|
| 今天 | 薄外套 | T恤 |

- 早晚温差大
- 外出戴口罩

`python3 fibonacci.py 10`
```

页面上必须显示为标题、表格、列表和行内代码，而不是原始 Markdown 文本。

### 3. Mention 渲染

支持：

- `@产品经理`
- `@Alice`
- `@user`
- Agent 选择器发送给某个 Agent 时，也应在消息里有明确视觉提示。

展示：

- 已匹配 Agent：黄色或浅色 mention chip，例如 `@产品经理`。
- user：中性 mention chip。
- 未匹配 mention：保留文本或弱化 chip，不打断排版。

数据结构：

```ts
export type Mention = {
  type: 'agent' | 'user';
  id: string;
  label: string;
};

export type Message = {
  ...
  mentions?: Mention[];
};
```

存储策略：

- 优先在 `messages` 表增加 `mentions` JSON 字段。
- 创建消息时根据当前 Agent 列表解析并持久化。
- 渲染时可用 `mentions` 精准匹配；没有 metadata 时 fallback 到文本解析。

输入体验：

- 输入 `@` 时出现轻量建议菜单。
- 候选包括当前 Agent 和 `user`。
- 支持键盘上下、Enter/Tab 选择、Esc 关闭。
- 不能破坏中文输入法组合态。

### 4. Thread

用户故事：

- 用户点击某条消息的“回复”。
- 右侧打开 Thread 面板。
- 面板顶部展示根消息。
- 用户可以在 Thread 里继续回复。
- Thread 回复不作为完整消息出现在主频道。
- 主频道根消息下展示摘要：`3 条回复 · 最新 10:42`。
- 点击摘要可以重新打开 Thread。
- Thread 内消息同样支持 Markdown 和 mention。

数据模型：

```ts
export type Message = {
  id: string;
  channelId: string;
  senderName: string;
  content: string;
  agentId?: string;
  createdAt: string;
  threadRootId?: string;
  replyCount?: number;
  latestReplyAt?: string;
  mentions?: Mention[];
};
```

规则：

- 顶层消息：`threadRootId = null`。
- Thread 回复：`threadRootId = rootMessage.id`。
- 回复必须和根消息属于同一个频道。
- 回复一个回复时，自动归到原始 root，不做嵌套 Thread。
- 删除频道时，相关 Thread 回复随频道消息一起处理。

API：

- `GET /api/channels/:id/messages`
  - 默认只返回顶层消息。
  - 顶层消息带 `replyCount` 和 `latestReplyAt`。
- `GET /api/messages/:id/thread`
  - 返回 `{ root, replies }`。
- `POST /api/channels/:id/messages`
  - body 支持 `threadRootId?: string`。
  - 无 `threadRootId` 创建顶层消息。
  - 有 `threadRootId` 创建 Thread 回复。

WebSocket：

- 顶层消息继续推送 `message:new`。
- Thread 回复建议新增 `thread:message:new`。
- Thread 回复到达时：
  - 如果当前打开的是该 Thread，追加回复。
  - 同时更新主频道根消息的 `replyCount/latestReplyAt`。

### 5. 布局与视觉调整

基于截图，v0.9 需要顺手做以下视觉治理：

- 主聊天区优先，右侧 Agent 面板默认宽度控制在 300-340px。
- Thread 打开时，Thread 面板可以替代右侧 Agent 面板，避免三栏同时挤压。
- 左侧导航仍可保留黄色品牌色，但减少纯黑粗边框。
- 消息列表背景使用接近白色的安静底色。
- 消息 hover 时才出现 `转任务`、`回复`、`复制` 等操作。
- 消息行最大阅读宽度建议 900-1040px，超长 URL 自动换行。
- 表格容器横向滚动，不撑破主布局。
- 输入区高度更稳定，按钮中文化。
- Agent 卡片改为更紧凑的列表形态：
  - 名称 + runtime badge
  - 状态点 + 中文状态
  - stop 按钮只在 hover 或卡片操作区展示

移动端/窄屏：

- Thread 面板使用覆盖层或替换主聊天。
- 右侧 Agent 面板默认隐藏到抽屉。
- 输入框和发送按钮不能溢出。

## 风险与注意点

- Thread 是数据模型变化，必须同时改 server 和 Cloudflare。
- Markdown 依赖可能引入默认样式，必须用 scoped class 控制。
- mention 解析不要在 v0.9 做复杂权限和路由，先展示和持久化。
- 当前 UI 的像素风不必完全移除，但要降低边框和高饱和色对长文阅读的干扰。
- 不要只做“看起来像 Thread”的前端状态，Thread 必须持久化。
