# SwarmOS 项目流程模块拆分

本文档用于把当前项目先拆成稳定、可并行推进的开发模块，并且严格对齐前置需求文档中的技术路线、MVP 范围、双人分工、风险控制和验收标准。后续实现时，建议严格按模块推进，先完成契约与基础闭环，再逐步增加能力，避免一次性铺开导致返工。

## 0. 技术路线总判断

本项目的实施路线必须与前置需求文档保持一致，结论是：

- 以 `crewden` 作为 MVP 的主骨架进行二次开发
- `ruflo` 仅作为设计参考，不作为 MVP 的核心运行时
- 不采用“Crewden UI + Ruflo 内核”的拼装式路线
- 调度中台、审批流、锁控、超时和执行安全能力都要由 SwarmOS 自己掌控

这一点不是实现偏好，而是前置需求文档已经明确的技术路线结论。

## 1. 需求追踪总表

下面这一层是为了保证“模块拆分”不是泛泛而谈，而是可以直接追踪到前置需求文档中的明确条目。

### 1.1 从实施方案文档继承的要求

必须覆盖的内容包括：

- 以 `crewden` 为主骨架
- 复用 `crewden` 的前端、server、daemon、WebSocket、agent 生命周期管理
- 自己补齐 PRD 中缺失的核心能力
- 不把 `ruflo` 当作 SwarmOS 的底层强依赖
- 按并行开发边界拆分工作流
- 先做 MVP 闭环，再考虑高级能力

### 1.2 从双人分工文档继承的要求

必须覆盖的内容包括：

- A 负责人主导 shared、server、web
- B 负责人主导 daemon 与执行安全
- 高冲突文件必须指定唯一 owner
- 双方只在协议层强对齐，其余按 owner 范围开发
- 先统一 shared 契约，再并行推进

### 1.3 MVP 必须实现的内容

必须明确写入模块路线的 MVP 最小集是：

1. `POST /api/v1/swarm/init`
2. 动态 agent runtime config 实例化
3. thought log 事件流
4. daemon 文件锁
5. daemon 命令超时强杀
6. 高危动作人工审批闭环
7. UI 端到端演示路径

### 1.4 明确后置的内容

必须保持后置，不应混入 MVP 主线：

- 复杂 DAG 编排
- 多机 federated swarm
- 向量记忆 / RAG
- 完整 MCP 平台化
- 多租户与企业权限
- 高级成本治理
- 自学习和自治优化

### 1.5 必须写进风险与验收的内容

必须明确包含：

- 协议定义发散风险
- daemon 改造过深风险
- thought stream 来源不稳定风险
- 审批流状态机复杂度风险
- 过早引入 `ruflo` 复杂能力的风险
- swarm/init 连通性验收
- 文件并发写入安全性验收
- 2 分钟内全链路 demo 验收

## 2. 严格实施逻辑

这份文档不是功能拼盘，而是实施链路。每一模块都必须以前一模块的输出作为输入，模块之间不是任意并行，而是有先后解锁关系。

### 2.1 顺序原则

1. 先定技术路线和边界，再开始改代码。
2. 先定 shared 契约，再让 server、daemon、web 进入实现。
3. 先让中台和 daemon 形成执行闭环，再做前端可视化。
4. 先完成安全控制，再做联调与测试。
5. 先完成 MVP 闭环，再做文档收口与后续扩展。

### 2.2 为什么不能随意拆

- shared 不稳定，所有消费方都会返工。
- 中台没有 swarm/init、事件流和运行态，就没有调度对象。
- daemon 没有锁、超时和审批，执行层就不安全。
- 前端没有真实事件源，UI 只能停留在演示层。
- 测试必须依附稳定链路，否则会和实现一起频繁变更。

## 2. 总体开发顺序

推荐的实现顺序如下：

1. 需求与边界确认
2. 协议与数据模型定义
3. 中台调度骨架搭建
4. Daemon 安全执行能力
5. 前端观测与审批界面
6. 联调与端到端验证
7. 文档收口与后续扩展规划

这个顺序的核心原则是：先把系统能说清楚，再把系统能跑起来，最后再把系统做得更稳。

## 3. 模块划分

### 模块 0：需求基线、技术路线与开发边界

目标：把 PRD、实施方案、双人分工说明统一到一套执行口径里，并明确本项目的技术路线只能是“Crewden 主骨架 + Ruflo 参考”。

前置依赖：无。

主要产出：

- 明确 Crewden 作为主骨架
- 明确 Ruflo 只做设计参考
- 明确 MVP 只做什么、不做什么
- 明确 A/B 两人的 owner 边界
- 明确高冲突文件和禁止跨界修改范围
- 明确第一阶段验收目标

完成后产出：

- 技术路线结论
- MVP 边界
- owner 边界
- 风险与里程碑
- 首批验收口径

依赖关系：无。

解锁下一步：只有当路线和边界确认后，才能开始 shared 契约设计。

建议先做的检查：

- 需求文档中的 MVP 目标是否一致
- 是否存在必须提前锁定的字段或状态机

### 模块 1：协议与数据模型

目标：定义系统后续所有模块共享的契约，并把前置需求文档里明确提到的字段、事件、状态和错误模型全部落到 shared 层。

前置依赖：模块 0。

主要内容：

- `swarm/init` 请求与响应
- `channel_id`、`swarm_id`、`agent_count`
- agent runtime 配置结构
- `agent_id`、`role`、`model`、`system_prompt`、`allowed_tools`
- thought log 事件
- `thought_log`
- approval 请求与结果
- `approval:requested`、`approval:resolved`
- 审批状态机 `planned -> risk_detected -> awaiting_approval -> approved/rejected -> executing -> finished`
- action 请求与结果
- `DaemonActionRequest`、`DaemonActionResult`
- lock 状态
- `waiting_lock`、`lock:update`
- 文件锁状态机 `unlocked -> locked -> released`
- timeout 错误结构
- `TimeoutErrorPayload`
- risk level 与运行状态枚举
- `awaiting_approval`、`timed_out`、`cancelled`

完成后产出：

- shared types
- schema / validation
- 状态机定义
- 事件名与状态名统一
- 供 server、daemon、web 消费的稳定契约

为什么要先做这一层：

- 中台、Daemon、前端都依赖它
- 先定协议，后续开发才能并行
- 避免每个模块自己定义一套字段

交付结果：

- 可复用的 shared types
- 可复用的 schema / validation
- 初版状态机说明
- 供 server、daemon、web 共同消费的稳定契约

解锁下一步：模块 1 完成后，server、daemon 和 web 才能按同一契约实现。

### 模块 2：中台调度骨架

目标：让系统具备“创建 swarm、管理 agent、接收执行结果、广播状态”的能力，并承担前置需求文档中定义的中台控制面职责。

前置依赖：模块 1。

主要内容：

- `POST /api/v1/swarm/init`
- dynamic agent runtime config 写入
- swarm/session 存储结构
- agent runtime 注册与唤醒
- thought log 广播
- browser / websocket 事件推送
- approval store
- action orchestration
- `waiting_lock` 重试队列
- 风险动作策略与审批挂起恢复

完成后产出：

- swarm/session 运行态
- 中台调度入口
- thought / approval / action / lock 的中台处理
- 供前端订阅的事件推送

这个模块解决的是“谁在协调一切”的问题。

交付结果：

- 中台可创建一个 swarm
- 中台可维护多个 agent 的运行态
- 中台可把关键事件推送给前端
- 中台可消费 daemon 返回的 `waiting_lock`、`awaiting_approval`、`timed_out`

解锁下一步：只有中台能够稳定创建 swarm 并收敛状态流转后，daemon 的安全执行才有完整接入面。

### 模块 3：Daemon 安全执行

目标：让本地执行具备可控性，避免并发写入、无超时、危险命令裸跑，并满足前置需求文档里对 daemon 的安全执行终端定位。

前置依赖：模块 1。

主要内容：

- 文件排他锁管理
- 文件路径规范化
- 独占锁表
- 锁释放通知
- action 执行抽象
- 文件读写、命令执行、目录操作统一走 action 接口
- 命令超时强杀
- 默认 60 秒 timeout
- 高危动作本地二次拦截
- 结构化结果回传
- thought / activity 输出采集

完成后产出：

- 文件锁管理器
- 统一 action 执行器
- timeout kill switch
- 风险动作二次拦截
- 结构化执行结果

这个模块解决的是“执行会不会失控”的问题。

交付结果：

- 同一文件并发写入能返回 `waiting_lock`
- 命令可按默认 60 秒超时处理
- 高危动作可返回 `awaiting_approval`
- daemon 最终必须能回传结构化执行结果供中台消费

解锁下一步：只有当 daemon 的执行安全链路稳定后，前端才能接入真实执行态展示。

### 模块 4：前端观测与审批界面

目标：让用户能看见 agent 的思考、等待、执行和审批过程，并严格匹配前置需求文档中的可观测性要求。

前置依赖：模块 1、模块 2、模块 3。

主要内容：

- thought stream 折叠视图
- 审批卡片与 approve / reject 操作
- 锁等待状态展示
- 执行状态时间线
- swarm 初始化入口或调试入口
- 将 thought log 与普通聊天消息分离展示

完成后产出：

- thought stream UI
- approval UI
- lock / timeout / running 状态展示
- 调试或初始化入口

这个模块解决的是“用户如何理解系统正在做什么”的问题。

交付结果：

- 关键事件不再混在普通聊天消息里
- 审批流可直接操作
- 执行状态对用户可见
- 能展示 swarm / approval / lock / timeout 的状态变化

解锁下一步：只有 UI 能稳定消费真实事件后，联调与端到端验证才有意义。

### 模块 5：联调与端到端验证

目标：把协议、中台、Daemon、前端连成闭环，并覆盖前置需求文档中明确提出的验收路径。

前置依赖：模块 4。

主要内容：

- mock 事件联调
- 真实 websocket 联调
- 文件锁冲突测试
- 超时 kill 测试
- 审批流测试
- MVP 演示路径验证
- 2 分钟内完成“创建 utils.py、写入冒泡排序、执行脚本、回显结果”的全链路演示

完成后产出：

- 可复现的全链路演示
- 并发锁冲突测试结果
- timeout 测试结果
- 审批流测试结果
- MVP 验收记录

建议验证的最小闭环：

- 用户输入需求
- 中台初始化 swarm
- agent 申请写文件或执行命令
- daemon 返回结果或挂起状态
- 前端展示 thought / approval / lock / result

解锁下一步：只有联调闭环稳定并且验收可复现，文档收口才有意义。

### 模块 6：文档收口与扩展预留

目标：把当前 MVP 的实现边界、协议和后续方向固化下来，并同步前置需求文档中要求的实施说明、联调说明和风险记录。

前置依赖：模块 5。

主要内容：

- API 文档
- 状态机文档
- 联调说明
- 已知风险清单
- 后续扩展路线
- MVP 验收映射与责任人记录

完成后产出：

- 最终实施说明
- 联调手册
- 风险清单
- 后续迭代入口

这个模块可以放到最后做，但不应该缺席。

## 4. 推荐的阶段推进方式

### 第一阶段：先定契约

先完成模块 0 和模块 1。

原因：

- 这是所有后续开发的基础
- 只有协议稳定，才能让多人并行

### 第二阶段：打通执行闭环

并行推进模块 2 和模块 3。

原因：

- 中台负责调度
- Daemon 负责执行
- 两者合起来才能形成真正闭环

### 第三阶段：补齐可见性

推进模块 4。

原因：

- 功能即使跑通，用户也需要看得懂
- thought、approval、lock 状态是 MVP 体验核心

### 第四阶段：联调与收口

推进模块 5 和模块 6。

原因：

- 先验证稳定性，再整理文档
- 便于后续继续迭代更复杂能力

## 5. 两人协作建议

如果继续按双人开发推进，必须沿用前置需求文档中的分工原则：

- A：shared、server、web 的主负责人
- B：daemon 与执行安全的主负责人
- 模块 0、5 由两人共同确认和收口

这样拆分的好处是：

- 协议先收敛
- 中台与执行层边界清楚
- 前端只消费稳定事件
- 测试在功能成型后再集中收口

## 7. 不做什么

为了严格保持与前置需求文档一致，本项目当前不把以下内容纳入 MVP 主线：

- 复杂 DAG 编排
- 多机 federated swarm
- 向量记忆 / RAG
- 完整 MCP 平台化
- 多租户 / 企业权限
- 高级成本治理
- 自学习与自治优化

## 8. 当前建议的起步任务

现在不要急着大面积实现功能，建议先做下面三件事：

1. 把 shared 协议字段定稿
2. 把 swarm/init 的输入输出定稿
3. 把文件锁、审批、超时三个状态机的最小版本画出来

只要这三件事稳定，后续就可以按模块逐个实现，不会乱。
