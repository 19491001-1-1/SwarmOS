# v1.4 — Review & Acceptance

> **给 Claude Code / Codex 的执行指令：** 实现本文档描述的 v1.4。按 TDD 工作：先写失败测试，再实现，再跑测试确认通过。完成后运行 `pnpm verify`；如果触及 Cloudflare Worker，也必须运行 `pnpm --filter @mini-slock/cloudflare exec wrangler deploy --dry-run`。如果触及 Web 构建，也运行 `VITE_API_BASE=https://xoxiang-hub.xingke0.workers.dev pnpm --filter @mini-slock/web build`。

## 背景

v0.x 的 Task Board 只有状态流转，v1.3 让 Agent 能推进工作。但真正的团队闭环需要“完成质量”：做完必须有 evidence、reviewer 可以 approve/request changes，用户或负责人可以验收。

## 目标

- 任务进入 `in_review` 时必须带 evidence 或说明。
- reviewer 可以 approve 或 request changes。
- request changes 会把任务退回 `in_progress` 并保留返工意见。
- done 任务展示 evidence 和 acceptance checklist。
- 高风险任务默认 executor 和 reviewer 不能是同一个 Agent。

## 非目标

- 不做复杂审批矩阵。
- 不做法务/财务等真实合规流程。
- 不做 PR 系统替代品；只记录 evidence link 和 review decision。
- 不做多级 review。

## 数据模型

```ts
type ReviewStatus = 'requested' | 'changes_requested' | 'approved' | 'cancelled';

type TaskReview = {
  id: string;
  taskId: string;
  requesterAgentId?: string;
  reviewerAgentId?: string;
  status: ReviewStatus;
  evidence: string[];
  checklist: Array<{
    label: string;
    checked: boolean;
  }>;
  comment?: string;
  createdAt: string;
  updatedAt: string;
};

type ReviewDecision = {
  id: string;
  reviewId: string;
  reviewerAgentId?: string;
  decision: 'approve' | 'request_changes';
  comment: string;
  createdAt: string;
};
```

扩展 `TaskContext`：

```ts
type TaskContext = {
  reviewerAgentId?: string;
  evidence?: string[];
  acceptanceChecklist?: string[];
  reviewIds?: string[];
  reviewNotes?: string[];
};
```

## 实现任务

### Task 1：Review API

Server 和 Cloudflare 增加：

```http
POST /api/tasks/:taskId/reviews
GET  /api/tasks/:taskId/reviews
POST /api/reviews/:reviewId/approve
POST /api/reviews/:reviewId/request-changes
```

Internal Agent API 同步增加：

```http
POST /internal/agent/:agentId/tasks/:taskId/reviews
GET  /internal/agent/:agentId/reviews
POST /internal/agent/:agentId/reviews/:reviewId/approve
POST /internal/agent/:agentId/reviews/:reviewId/request-changes
```

规则：

- 创建 review 时任务状态变为 `in_review`。
- approve 后任务可进入 `done`。
- request changes 后任务回到 `in_progress`。
- high risk task 如果 reviewer 和 owner 相同，返回 400，除非 payload 显式 `allowSelfReview: true` 且 reason 存档。

### Task 2：Reviewer recommendation

复用 v1.0 role/capability：

- 根据任务 context 推荐 reviewer。
- reviewer 优先选择具备 `review`、`qa`、`product` 或相关 capability 的 Agent。
- 找不到 reviewer 时允许用户手动指定。

### Task 3：CLI

新增：

```bash
xoxiang review list
xoxiang review request <taskId> --reviewer <agentName> --evidence "..." --check "..."
xoxiang review approve <reviewId> --comment "..."
xoxiang review request-changes <reviewId> --comment "..."
```

要求：

- `review list` 默认列出分配给当前 Agent 的 pending reviews。
- evidence 支持多次传入或 JSON array。

### Task 4：Bridge prompt

更新 Agent 指令：

- 完成任务前检查 acceptance criteria。
- 不能只有“已完成”四个字，必须提交 evidence。
- 需要 review 的任务进入 `in_review`。
- reviewer 要逐条核对 checklist，不要无脑 approve。
- request changes 必须具体说明改什么。

### Task 5：Web UI

Task detail 增加 Review 区块：

- evidence list
- acceptance checklist
- reviewer
- review status
- approve / request changes 操作
- review history

Task Board：

- `IN REVIEW` 卡片展示 reviewer 和 evidence count。
- `DONE` 卡片展示 accepted 标记。

## 手动验证

1. 创建一个有 acceptance criteria 的任务。
2. 工程师提交 evidence 并 request review。
3. 测试 Agent request changes。
4. 工程师补充 evidence 后再次 request review。
5. 测试 Agent approve。
6. 确认任务进入 done，且 evidence/checklist 可见。
